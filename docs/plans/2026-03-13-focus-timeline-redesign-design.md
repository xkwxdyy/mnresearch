# Focus 驾驶舱与问题时间线重设计

**Goal:** 把 `mnresearch` 里当前偏“会后总结器”的专注会话，重构成真正服务科研推进的 `Focus` 驾驶舱；同时把问题页升级成以“研究时间线”为主舞台的工作台。

**Core Judgment:** 现在的问题不是“专注会话字段还不够多”，而是建模方向偏了。科研工具的核心不是计时，不是会话报表，也不是逼用户填“稳定度/置信度”这种低收益字段；核心应该是帮助用户在研究过程中快速记录有信息含量的推进痕迹，并把这些痕迹组织成问题的演化时间线。

**Date:** 2026-03-13

---

## 1. 已确认的设计结论

### 1.1 `Focus` 不是专注会话对象

- 不再把 `Focus` 设计成一个需要 `start / pause / resume / complete` 生命周期的独立对象。
- 不再把“开始专注、暂停专注、恢复专注、结束专注”当成核心研究数据。
- `Focus` 的新角色是一个常驻研究驾驶舱，而不是一个 session 容器。

结论：

- 删除“专注会话”作为核心产品概念。
- 保留 `Focus` 主视图，作为当前研究上下文的工作入口。

### 1.2 时间的角色要弱化

- 用户不要时长统计，不要番茄钟，不要报表式“我今天专注了多久”。
- 时间只保留一个作用：给研究事件排序，形成时间线。
- 真正重要的是“发生了什么研究输入”，不是“持续了多长时间”。

结论：

- 时间线只记录 `触发点` 和 `输入点`。
- 时间不再作为 KPI 或主分析维度。

### 1.3 问题页的主舞台必须是时间线

- 问题页不该再以“若干结构 tab 平铺堆放”为唯一主体验。
- 真正的研究推进应该先能看到这个问题是怎么演化到今天的。
- `判断 / 策略 / 障碍 / 认识 / 动作` 是结构化沉淀层，不该抢走主舞台。

结论：

- 问题页默认主舞台改成“问题时间线”。
- 其他结构区退到沉淀层和详情层。

### 1.4 快速输入必须极轻

- 用户明确要求：一个输入框，四个类型按钮。
- 类型固定为：
  - `进展`
  - `认识`
  - `卡点`
  - `下一步`
- 初始状态不预选任何类型。
- 用户输入内容后，点击某个类型按钮，直接按该类型提交。
- 不要额外“保存”按钮。
- 提交后输入框清空，并回到未选中状态。

结论：

- 采用“单输入框 + 四按钮”的极轻录入方式。
- 录入对象先是轻量时间线事件，而不是正式研究实体。

### 1.5 快输入默认只生成轻量时间线事件

- 用户不希望在灵感刚冒出来时就被迫决定“这是不是正式认识 / 正式障碍 / 正式策略”。
- 快输入的第一目标是保留研究火花，而不是立刻规范化建模。

结论：

- `进展 / 认识 / 卡点 / 下一步` 四类快输入，默认都只生成轻量时间线事件。
- 后续从时间线事件进入详情，再做：
  - 编辑
  - 删除
  - 升格为正式对象

### 1.6 “策略”与“动作”必须拆开

- 当前 `strategy` 太宏观，但又被拿来承载可执行下一步，角色混乱。
- 策略应该是研究方向。
- 动作应该是可执行计划。

结论：

- 新增独立的“计划 / 动作”层。
- `策略` 继续作为宏观推进路线。
- `动作` 作为执行层，可挂在问题下，也可挂在策略下。
- 问题层提供动作汇总视图。

### 1.7 置信度 / 稳定度直接删

- 用户明确判断：
  - `认识稳定度`
  - `总结置信度`
  - `继续点置信度`
  这些字段当前收益太低，录入负担太高。

结论：

- 直接删除上述字段。
- 不做默认隐藏，不做高级模式，先砍。

### 1.8 没投入使用的“大焦点笔记”占位必须移除

- 当前界面里“焦点笔记 / 后续接 MN 卡片 / 暂未接入”这块没实际价值，还占主要视觉空间。
- 用户已经明确吐槽这一块。

结论：

- 移除当前 Focus 面板中这块未接入的大占位。
- 如果未来需要扩展，重新以轻量入口方式设计，而不是预留一整块空壳。

### 1.9 子问题时间线默认独立，可切换并入母问题

- 子问题应该有自己的完整演化线。
- 母问题默认不应被子问题事件刷屏。
- 但需要一个总览模式，让用户看到整个研究家族按时间展开的全局演化。

结论：

- 默认：母问题时间线只显示母问题自己的事件。
- 提供开关：开启后，把子问题时间线事件按真实时间顺序插入母问题时间线。
- 插入事件必须标明来源子问题。

### 1.10 自动记录必须有信息含量

- 用户允许自动记录，但反对“修改了问题”“更新了策略”这种废话日志。
- 自动记录必须能回答“改成了什么 / 新增了什么 / 发生了什么有内容的变化”。

结论：

- 自动记录示例：
  - `问题表述改为：……`
  - `新增策略：……`
  - `策略「A」改为「B」`
  - `从时间线事件升格为认识：……`
- 禁止无信息量日志：
  - `修改了问题`
  - `进行了编辑`
  - `更新了内容`

### 1.11 时间线详情用按需弹窗，不做常驻详情栏

- 用户明确选择：点击某条时间线事件后，再弹出详情 modal。
- 不做右侧常驻详情栏。
- 不做下方常驻展开区。

结论：

- 时间线主列表保持简洁。
- 事件详情按需打开。
- 详情里再做编辑 / 删除 / 升格。

### 1.12 时间线列表要有“时间瀑布”感

- 时间必须放在最前面。
- 不要普通卡片列表那种“标题在前、时间在角落”的信息层级。
- 时间线条目采用两层结构：
  - 第一行：`时间 + 类型 + 摘要`
  - 第二行：`来源 + 关联对象`

结论：

- 时间是时间线视觉主轴。
- 第二行只放上下文，不抢摘要。

### 1.13 `当前聚焦` 只允许策略和动作

- 用户修正后的选择是：
  - `无`
  - `策略`
  - `动作`
- 不把判断、障碍、认识、文献塞进当前聚焦 selector。

结论：

- `当前聚焦` 候选只来自“当前绑定问题”下的策略与动作。
- 不允许跨问题乱选。

### 1.14 动作区保留，但默认收起

- `Focus` 里需要有动作区。
- 但动作区不能抢主舞台。
- 默认收起，展开后用卡片呈现。

结论：

- `Focus` 里保留可折叠的轻量动作区。
- 展开后每张动作卡默认提供：
  - `设为当前聚焦`
  - `改状态`
  - `查看详情`
- `改状态` 直接走小菜单，不再多跳一步。

### 1.15 动作状态采用五态

- 用户确认的最小状态集：
  - `queued`
  - `doing`
  - `waiting`
  - `blocked`
  - `done`

结论：

- 第一阶段动作状态直接采用这五态。

### 1.16 `进展` 不升格，但必须可单独筛出

- 用户明确要求：
  - `进展` 不升格成正式对象
  - `进展` 要能被单独筛出来
  - 这样方便对导师汇报

结论：

- `进展` 永远留在时间线里。
- 时间线过滤器必须支持只看 `进展`。

### 1.17 过滤器采用两个下拉

- 用户明确选择：

```text
类型: [全部 v]
来源: [全部 v]
```

其中：

- 类型：
  - `全部`
  - `进展`
  - `认识`
  - `卡点`
  - `下一步`
- 来源：
  - `全部`
  - `手动输入`
  - `系统记录`

结论：

- 类型和来源分开，不混成一个大筛选器。

### 1.18 Focus 与问题树是独立状态

- Focus 只会在点击左侧 `研究驾驶舱 / Focus` 按钮后显示。
- 左侧问题树继续独立承担导航。
- 左侧点别的问题，不自动改变 Focus 的绑定问题。

结论：

- `Focus` 有自己的绑定记忆。
- 默认回到上一次绑定的问题。
- 如果绑定问题不存在了，回退到空状态，不替用户瞎猜。

## 2. 目标信息架构

### 2.1 页面总布局

用户确认的布局如下：

```text
+------------------+----------------------------------+
| 左侧             | 右侧                             |
|                  |                                  |
| 问题树           | 这里二选一：                     |
| [研究驾驶舱按钮] | 1. 问题工作台                    |
|                  | 2. Focus 驾驶舱                  |
|                  |                                  |
+------------------+----------------------------------+
```

关键点：

- 左侧问题树始终保留。
- 在问题树上方新增 `研究驾驶舱 / Focus` 切换按钮。
- 只切右侧主内容区。
- 问题树继续承担全局导航作用，不能消失。

### 2.2 Focus 驾驶舱主结构

用户确认的主骨架如下：

```text
+------------------------------------------------------+
| 绑定问题：Q1                  当前聚焦：策略S2 [切换] |
+------------------------------------------------------+
| 快速输入                                               |
| [进展] [认识] [卡点] [下一步]                          |
+------------------------------------------------------+
| 问题时间线                                             |
| - ...                                                  |
+------------------------------------------------------+
```

绑定方式：

- 两级绑定：
  - 一级：绑定问题
  - 二级：当前聚焦对象（可选）
- 头部采用两个 selector：

```text
[绑定问题: Q1 v]   [当前聚焦: 策略S2 v]
```

### 2.3 时间线交互

时间线条目不应直接堆一排操作按钮。用户选择的是详情式操作。

推荐交互：

```text
时间线列表
  -> 先看内容
  -> 点一条
  -> 打开详情抽屉 / 详情面板
  -> 再做 [编辑] [删除] [升格]
```

好处：

- 主列表更干净。
- 不会变成“按钮墓地”。
- 升格路径更清晰。

## 3. 新的核心产品模型

### 3.1 `Focus` 的职责

`Focus` 只做三件事：

1. 绑定当前研究问题
2. 绑定当前聚焦对象
3. 提供常驻、极轻的研究输入入口

`Focus` 不负责：

- 会话开始 / 暂停 / 恢复 / 完成
- 时长统计
- 会后报表
- 大块空壳式笔记容器

### 3.2 问题时间线的职责

问题时间线是这个问题的研究演化主线。它应该统一承接：

- 问题创建
- 问题表述更新
- 判断新增 / 修改
- 策略新增 / 修改
- 障碍新增 / 修改
- 认识新增 / 修改
- 动作新增 / 修改
- Focus 快输入产生的轻量事件
- 时间线事件升格为正式对象

时间线不是：

- 操作日志收纳桶
- 专注计时历史
- 系统 debug log

### 3.3 新增“动作”层

建议新增独立实体，例如：

```js
actions: [
  {
    id: "action-1",
    questionId: "q1",
    strategyId: "s2", // 可空
    title: "先验证二维边界情形",
    description: "",
    status: "todo", // todo / doing / blocked / done
    sourceTimelineEventId: "event-7", // 可空
    createdAt: "...",
    updatedAt: "..."
  }
]
```

判断：

- 这是策略和执行之间缺失的那一层。
- 没这层，策略会继续变成“宏观方向 + 微观 todo”的混血怪物。

### 3.4 新增“时间线事件”层

建议新增顶层数组，例如：

```js
timelineEvents: [
  {
    id: "event-1",
    questionId: "q1",
    relatedEntityType: "strategy", // 可空
    relatedEntityId: "s2", // 可空
    eventType: "quick_note", // quick_note / question_updated / strategy_added / ...
    noteType: "insight", // progress / insight / obstacle / next_step，可空
    content: "二维这里真正卡住的不是估计，而是方向组织。",
    source: "focus_input", // focus_input / auto_record / promote
    sourceQuestionId: "q1", // 方便母子问题混流
    sourceQuestionTitle: "主问题标题",
    promotedEntityType: "", // 可空
    promotedEntityId: "", // 可空
    createdAt: "...",
    updatedAt: "..."
  }
]
```

关键原则：

- 时间线事件是一等公民。
- 它不是 `progressLog` 的换皮。
- 它必须承载研究内容，而不是机械动作。

## 4. 时间线记录规则

### 4.1 应记录什么

应记录：

- 有明确文本内容的研究输入
- 有明确变化结果的结构修改
- 对研究推进有解释力的升格动作

示例：

- `问题表述改为：把目标缩到有限扇区模型。`
- `新增策略：先拆二维 proof 中真正依赖平面几何的步骤。`
- `进展：selection rule 已被抽成可替换结构。`
- `认识：真正瓶颈是方向连续族组织，不是 size estimate。`
- `卡点：找不到一个足够小但不失真的反例模型。`
- `下一步：先做二维边界情形，再看能否上升到扇区模型。`
- `从时间线事件升格为动作：先验证二维边界情形。`

### 4.2 不应记录什么

不应记录：

- 会话生命周期
- 纯 UI 切换
- 没有内容的机械编辑
- 单纯“打开了某个面板”

禁止示例：

- `开始专注`
- `暂停专注`
- `恢复专注`
- `完成专注`
- `修改了问题`
- `更新了策略`

### 4.3 自动记录策略

自动记录允许存在，但必须遵守：

1. 只记录有内容差异的字段
2. 优先生成“新值摘要”
3. 不生成模板废话

建议：

- 问题表述修改：
  - `问题表述改为：……`
- 策略名称修改：
  - `策略由「旧名」改为「新名」`
- 动作新增：
  - `新增动作：……`

### 4.4 母问题与子问题混流

默认：

```text
母问题时间线 = 只看母问题自己的事件
```

开启开关后：

```text
母问题时间线 = 母问题事件 + 子问题事件（按时间混排）
```

混流事件必须带来源标记，例如：

```text
[来自子问题：端点反例最小模型]
认识：这里的失败机制来自尺度竞争，不是单点异常。
```

## 5. Focus 驾驶舱交互细则

### 5.1 快输入交互

用户确认的录入规则：

```text
[进展] [认识] [卡点] [下一步]
--------------------------------
| 单一输入框                     |
--------------------------------
```

规则：

- 初始状态不选中任何类型。
- 用户先输入内容，再点某个类型按钮。
- 点击按钮即提交。
- 提交后输入框清空。
- 提交后回到未选中状态。
- 不额外提供保存按钮。

这是一个“先写，再归类”的模式，比“先决定类型再写”更贴近真实思考。

### 5.2 绑定问题与当前聚焦对象

头部显示两个选择器：

```text
[绑定问题: Q1 v]   [当前聚焦: 策略S2 v]
```

含义：

- `绑定问题`
  - 决定时间线归属
  - 决定当前研究上下文
- `当前聚焦`
  - 只是给输入增加局部语境
  - 例如让一条“卡点”知道自己是在某个策略或某个动作下出现的

### 5.3 Focus 与问题工作台的关系

- `Focus` 是输入和推进视图。
- 问题工作台是结构化整理视图。
- 两者通过右侧主内容区切换，不通过新页面跳转。

这意味着：

- 用户可以常驻在 Focus 里推进
- 也可以回到问题工作台整理结构
- 左侧问题树始终是全局导航锚点

## 6. 对现有实现的直接改造建议

### 6.1 应废弃的现有概念

当前实现中的以下内容建议废弃或降级：

- `focusSessions`
- `progressLog`
- `summary-confidence`
- `edit-focus-confidence`
- `new-insight-stability`
- `edit-insight-stability`
- Focus 面板里的“大焦点笔记占位”

### 6.2 现有 `progressLog` 不能直接升级为新时间线

原因：

- 现有 `progressLog` 主要记录机械事件：
  - `started`
  - `paused`
  - `resumed`
  - `completed`
  - `item_linked`
- 这不是用户要的研究时间线。

建议：

- 新建独立的 `timelineEvents`
- 不要硬拿 `progressLog` 改名糊弄
- 可择机做一次有限迁移：
  - 仅保留少数真正有内容的旧记录
  - 机械记录直接丢弃

### 6.3 Focus UI 要从“浮动会话卡”重做

当前实现偏浮动面板 + 会话控制：

- 暂停
- 继续
- 完成并沉淀
- 删除会话

这套思路和本次设计方向冲突。

建议：

- 右侧主内容区新增真正的 `Focus` 页面模式
- 不再围绕“会话控制”组织 UI
- 围绕“绑定上下文 + 快速输入 + 时间线阅读”组织 UI

## 7. 推荐开发分阶段

### Phase 1: 先把骨架立正

- 新增右侧 `Focus` 视图模式
- 左侧问题树顶部新增 `研究驾驶舱` 切换按钮
- 移除现有 Focus 浮动会话控制思路
- 实现：
  - 绑定问题 selector
  - 当前聚焦 selector
  - 单输入框 + 四按钮
  - 问题时间线主列表

### Phase 2: 新数据模型落地

- 新增 `timelineEvents`
- 新增 `actions`
- 建立时间线详情面板
- 实现时间线事件到正式对象的升格路径

### Phase 3: 自动记录与混流

- 问题 / 判断 / 策略 / 动作的有信息量自动记录
- 子问题事件混入母问题时间线的开关
- 来源标记与过滤

### Phase 4: 清理旧模型

- 迁移或清理 `focusSessions`
- 迁移或清理 `progressLog`
- 删除置信度 / 稳定度相关 UI 与字段
- 删除未接入的大焦点笔记区域

## 8. 当前还未拍板的点

以下事项尚未在本轮讨论中最终确认，后续开发前应继续定稿：

- 时间线详情面板是右侧抽屉、下方展开区，还是 modal
- `当前聚焦` selector 是否允许跨问题选择对象，还是只能选当前绑定问题下的对象
- “动作”对象的最小字段集和状态枚举
- 时间线的默认排序是否支持“只看手动输入 / 只看自动记录 / 只看某一对象”过滤
- 问题工作台与 Focus 视图之间，是否要共享同一个“当前问题”上下文，还是允许 Focus 独立绑定其他问题
