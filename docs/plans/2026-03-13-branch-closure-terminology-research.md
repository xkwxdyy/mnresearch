# 分支闭环术语重设计调研

**Date:** 2026-03-13

**Goal:** 把 `mnresearch` 里围绕“策略分支 -> 子问题 -> 结果送回母问题”的整套术语，从当前偏抽象、偏黑话的表达，重设计成对小白更直观、但依然保留专家级研究语义的工作语言。

**Core Judgment:** 这次不是改一个词，而是在重写一条研究闭环的语言系统。`回流` 现在同时承担了动作、状态、结果归属和阶段信号四种职责。只改表层 label，不改周边术语，最后只会更乱。

---

## 1. 是否值得单独立文档

值得，而且必须单独立。

原因很直接：

1. `回流` 已经不是一个孤立文案，而是跨了数据结构、生命周期、仪表盘统计、侧栏提示、问题卡片、分支地图、弹窗表单和设计文档。
2. 这次改动的核心不是实现难，而是语义风险高。词一旦定歪，后面所有状态名、按钮名、提示语都会跟着发臭。
3. 这个需求天然需要分阶段推进：
   - 先定核心隐喻和术语族。
   - 再定每个界面位置到底说什么。
   - 最后才决定是否连内部字段名一起重构。
4. 用户明确要求“我是小白，但要达到专家级效果”。这意味着术语必须兼顾：
   - 一眼能懂下一步要做什么；
   - 不牺牲研究结构表达力；
   - 能反向教会用户专家是怎么推进研究的。

结论：

- 本文档作为这次术语重构的主调研与任务追踪文档。
- 暂时不写 implementation plan。
- 等关键术语确认后，再决定是否补一份实现计划文档。

---

## 2. 当前系统里，“回流”到底在指什么

当前代码里，“回流”至少在表达 4 件不同的事：

1. **一个动作**
   - 用户对某个子问题点击“记录回流”。
   - 含义是：把局部分支结果显式写回主问题推进。

2. **一个状态**
   - `pending / partial / fed_back`
   - 前端显示为：`待回流 / 部分回流 / 已回流`

3. **一个结果摘要**
   - `feedBackSummary`
   - 前端显示为：`回流摘要`

4. **一个生命周期信号**
   - 子问题已 `resolved` 但未 `fed_back` 时，问题生命周期阶段会变成 `待回流`。
   - 驾驶舱和问题总览也会把它统计为一个待处理风险。

问题在于：这四层语义混在一起，导致用户看到“回流”时，不知道它更像：

- “把结果写回去”这个动作；
- “这个分支是否已经完成闭环”的状态；
- “这条分支主要服务母问题什么部位”的贡献关系；
- 还是“研究收尾”这一阶段。

这就是它显得抽象的根源。

---

## 3. 已确认的真实功能链

当前仓库已经有一条完整但语言偏硬的分支闭环链路：

1. 在母问题下创建策略。
2. 策略可以继续留在策略层，也可以“升格为子问题”或“关联已有子问题”。
3. 一旦形成子问题，系统会记录：
   - 这个分支扮演什么角色；
   - 它主要给母问题贡献什么；
   - 它来自哪条策略或哪条问题。
4. 子问题推进后，用户可以通过显式表单记录“结果送回母问题”的状态和摘要。
5. 这套状态会反过来影响：
   - 子问题卡片的分支状态；
   - 侧栏树提示；
   - 分支地图；
   - 驾驶舱分支健康统计；
   - 问题生命周期阶段与建议下一步。

所以，系统实际在做的是：

`策略发起分支 -> 子问题承接深入 -> 结果归位主线 -> 主问题继续推进`

问题不在功能缺失，而在这条链的语言没有把动作讲透。

### 4.4 UI 文案层

- 侧栏：`源自策略`、`待补关系`、`待回流`

### 4.5 文档层

- 这是一个跨层术语重构。
- 第一阶段只改 UI 词，也必须同步设计文档，否则文案和实现会马上分叉。

## 5. “回流”为什么不够好

### 5.1 它像一个系统内部词，不像用户操作词

“回流”对设计者来说能懂，但对用户来说，第一反应通常不是：

- 要把什么送回去？
- 送回到哪？
- 送回后会发生什么？

它更像一个系统内部状态名，而不是一个能指导动作的词。

### 5.2 它没把“主线 / 母问题 / 作用点”讲出来

当前系统其实想表达的是：

- 子问题不是独立收集箱；
- 子问题跑出的结果，要重新影响母问题、策略、判断或障碍。

但“回流”没有把目标对象说出来，所以用户仍然要脑补。

### 5.3 它和“贡献类型”混在一起了

现在界面里同时有：

- `回流状态`
- `这条分支解决后主要回流什么`

第二句其实在问的是“完成后对主问题起什么作用”，并不是“回流到哪个动作状态”。

也就是说：

- 一个词被同时拿来表示“送回动作”和“贡献方向”。
- 这会让表单非常拧巴。

### 5.4 它对小白不够教学

你要的是“科研最强大脑”，不是术语堆砌器。

真正好的词应该做到：

- 小白看到后知道下一步动作；
- 用着用着自然学会专家怎么处理研究分支；
- 不需要在脑子里二次翻译。

“回流”现在做不到这一点。

## 6. 这次不该只改“回流”，应该一起重写的术语邻居

如果核心词改了，下面这些词最好一起动：

| 当前词 | 当前角色 | 为什么可能也要改 |
| --- | --- | --- |
| 回流状态 | 分支结果状态 | 核心词直连 |
| 记录回流 | 按钮文案 | 动作词必须跟着换 |
| 保存回流 | 提交按钮 | 同上 |
| 回流摘要 | 结果摘要字段 | 用户要知道自己在写什么 |
| 待回流 | 生命周期阶段 | 阶段词要和动作词一致 |
| 已回流 | 完成态 | 同上 |
| 这条分支解决后主要回流什么 | 贡献类型字段 | 这句本身就别扭，必须重写 |
| 分支关系 | 卡片标题 | 若核心隐喻从“分支”转成“研究支线/子任务”，这里可能也要变 |
| 源自策略 / 来源策略 | 来源说明 | 需要与新的闭环语言统一 |
| 待补关系 / 悬空子问题 | 数据缺口提示 | 也要和新的主隐喻一致 |

## 7. 推荐的术语方向

这里不是最终拍板，只是现阶段调研后的方向判断。

### 方向 A：`接回主线` 系

示例：

- `待接回主线`
- `部分接回主线`
- `已接回主线`
- `记录接回`
- `接回摘要`
- `这条分支完成后主要支撑什么`

优点：

- 小白最容易懂。
- 自动把目标对象说出来了：不是“回流到空气里”，而是“接回主线”。
- 很符合科研推进的心智：主线、支线、重新汇回。

缺点：

- 偏口语，学术感没那么强。
- 在某些窄 UI 里字会偏长。

### 方向 B：`结果归位` 系

示例：

- `待归位`
- `部分归位`
- `已归位`
- `记录归位`
- `归位摘要`
- `这条分支完成后主要作用于哪里`

优点：

- 比“回流”更具体，但比“接回主线”更短。
- 有一种“把局部结果放回整体结构”的专家味道。
- 比较适合科研系统，不会太像任务管理软件。

缺点：

- 对完全小白来说，第一眼仍然需要半秒理解。
- 如果不配合说明文案，动作感不如“接回主线”强。

### 方向 C：`纳入主问题推进` 系

示例：

- `待纳入推进`
- `部分纳入推进`
- `已纳入推进`
- `记录纳入结果`

优点：

- 非常精确。
- 直接表达“结果已经被主问题采用”。

缺点：

- 太长。
- UI 里会显得笨重，不够像产品语言。

### 当前推荐

**我目前最推荐方向 A：`接回主线`，方向 B：`结果归位` 作为次选。**

理由：

1. 你明确说“回流太抽象”，那就别再换成另一个抽象词。
2. 你又要求“小白也能做出专家效果”，那术语本身就该承担教学作用。
3. `接回主线` 天生带有路径感和结构感，能把分支研究的闭环讲明白。
4. 如果后续觉得太口语，可以在局部用混合式表达：
   - 大按钮用 `接回主线`
   - 统计和摘要里用 `归位`

## 8. 我建议的第一阶段改法

不要一上来就动内部字段名。

推荐分两层：

### 第一层：先改对用户可见的语言

- 所有 UI 文案
- 所有设计文档
- 所有说明文字
- 所有阶段名和按钮名

这样收益最大，风险最小。

### 第二层：确认稳定后，再决定要不要改内部键名

当前内部键名还在用：

- `feedBackStatus`
- `feedBackSummary`
- `fed_back`

这些名字很丑，但它们现在是稳定状态源。

如果现在边定术语边改内部键名，会把风险同时抬高到：

- 语言歧义风险
- 数据兼容风险
- 逻辑回归风险

这不划算。

我的建议：

- 第一阶段只重构外部语言。
- 第二阶段如果术语稳定，再考虑把内部键名迁成更干净的 `closureStatus` / `integrationStatus` 一类。

## 9. 当前不确定点

这些点需要用户逐个确认，但不该一次性全砸过去：

1. 你想把这条闭环主隐喻做成偏口语教学型，还是偏研究系统型？
2. 这条结果到底应该强调“回到主线”，还是强调“纳入母问题”，还是强调“写回结构”？
3. “分支”这个词要不要一起降门槛，比如换成“研究支线 / 深挖子题 / 支线问题”？
4. “策略”这个词在这套链路里对你来说是否已经够直观，还是也偏抽象？
5. 这次是否只做术语与界面改造，还是要顺带优化交互顺序，例如：
   - 子问题标记已解决时自动引导做结果归位；
   - 分支卡顶部直接显示“下一步该接回什么”；
   - 分支地图里直接突出“哪些支线还没接回主线”。

## 10. 技术审计补充

这轮代码审计又确认了 6 个不能忽略的事实：

1. **“回流”现在有双真源**
   - `question.branchMeta.feedBackStatus/feedBackSummary`
   - `branchLink.status/note`
   两边都在表达闭环状态，但统计与摘要并不完全统一。

2. **摘要层实际上更信 `question.branchMeta`**
   - `normalizeQuestion()` 会默认把每个问题的 `feedBackStatus` 设成 `pending`。
   - 这让 `getQuestionBranchSummary()` 对 `branchLink.status` 的回退几乎失效。

3. **全局统计和局部统计口径并不一致**
   - 某些地方看 `childSummary.feedbackStatus`
   - 某些地方看 `branchLink.status`
   这意味着后续只改文案还不够，至少要顺手统一状态解释。

4. **普通 `+ 子问题` 会制造“树上有孩子，但研究关系没建”的悬空分支**
   - 它写了 `branchMeta`
   - 却不写 `branchLink`
   所以这条链天然容易出现“看起来像分支，实际上没接线”的脏状态。

5. **记录闭环结果时没有时间线事件**
   - 状态改了
   - 但研究时间线没有留下“这条支线结果已并回主线”的痕迹
   这对“科研最强大脑”是明显损失。

6. **存在交互与模型不完全对齐的边角**
   - “关联已有问题”允许把非子问题接到策略上；
   - 但“记录回流”弹窗又只接受 `isChild` 问题。
   这说明当前模型还没完全想透“闭环动作到底只属于子问题，还是也属于被关联的问题”。

结论：

- 这次先定术语是对的。
- 但术语一旦定下来，后面极可能要顺手清一次状态口径和交互边界。

## 12. 第一轮结论

这次该改的不是“回流”这个词本身，而是整个“分支闭环语言”。

**当前判断：**

- 要单独立文档。
- 先定语言，再动代码。
- 第一阶段优先改用户可见术语，不碰内部字段。
- 核心术语推荐先围绕 `接回主线` 和 `结果归位` 两个方向做确认。
